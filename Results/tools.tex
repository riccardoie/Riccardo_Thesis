\chapter{NVOL in METIS and Supporting Tooling}\label{chap:NVOL tooling}
Chapters~\ref{cv_calculation} and~\ref{chapter5} developed NVOL, our partition-specific communication-volume objective function, and showed how its integrated in the refinement phase. This chapter turns from theory to practice: it describes how NVOL was integrated into METIS, and the Python tooling developed alongside it to analyze and visualize the resulting partitions. The implementation and tools described here are used to produce the results presented in Chapter~\ref{chap:results}.

The first section starts with a general description of METIS and how to run it. Section~\ref{sec: implementing NVOL} then details how the NVOL algorithm was integrated into METIS's C source. The second section presents the tools developed to help analyze the results. Together they form the complete experimental pipeline: the modified METIS produces the partitions, and the Python tools turn those partitions into the analyses we report in Chapter~\ref{chap:results}.


\section{METIS} \label{sec:Metis very general description}
For this thesis we used METIS as our multilevel $k$-way partitioner. The METIS framework is open source\footnote{\url{https://github.com/KarypisLab/METIS}} and developed at the Department of Computer Science \& Engineering at the University of Minnesota~\cite{KarypisGeorge1997MASP}. Its quality and efficiency have made it widely used. The software, written in C, ships with several programs for partitioning a graph, partitioning a mesh, computing a fill-reducing ordering of a sparse matrix, and converting a mesh into a graph. Given the scope of this thesis, we worked with the graph partitioning programs. Since most of the logic behind the multilevel $k$-way algorithm has already been covered in Chapters~\ref{chap:multilevel-kway}--\ref{chapter5}, we refer the reader to those chapters. Using METIS as the base, we implemented our own objective function, adapting the C source to our needs. This adapted version of METIS with NVOL has been made public\footnote{\url{https://github.com/riccardoie/METIS-with-NVOL}}. The details are presented in Section~\ref{sec: implementing NVOL}.

\subsection{Inputs}
METIS requires three inputs: a graph file, the number of partitions, and the objective function. A few optional flags are also available to fine-tune the algorithm.

The graph file must follow a specific structure. The first line is a header whose first two values are the number of vertices and the number of unique edges in the graph. The header may also contain two additional parameters specifying whether the graph has vertex/edge weights and whether it has multiple weights per vertex or edge. Further details can be found in the official manual\footnote{\url{https://github.com/KarypisLab/METIS/blob/master/manual/manual.pdf}}. All graphs used in this work are single-weight, with unit weights on both edges and vertices. A list of them can be found in Table~\ref{tab:heart_dataset} and Table~\ref{tab:suitesparse}. Note, however, that NVOL also supports graphs with single (non-unit) weight per vertex.

The remaining lines of the graph file describe the graph's connectivity: each line corresponds to a vertex, and its entries are the ids of the vertices it is connected to.

\subsection{Output}
Once the algorithm terminates, METIS produces a partition file. This text file contains $n$ lines, where $n$ is the number of vertices in the graph, and each line holds a single integer in the range $[0, k-1]$ indicating the partition the corresponding vertex was assigned to. The partition files are named graphName.part.$k$, where $k$ is the number of partitions. 

\subsection{Implementing NVOL} \label{sec: implementing NVOL}
Implementing NVOL required extending METIS's existing infrastructure. We first defined a new volume-based objective function called NVOL, which can now be passed as an argument when running the program. Since it is volume-based, NVOL reuses the coarsening and initial partitioning code of the total volume objective function (VOL) (see Sections~\ref{sec:coarsening}--\ref{sec:initial_partitioning}); our extensions are confined to the refinement stage (see Section~\ref{sec:uncoarsening}).

Before describing those extensions, we briefly review the data layout they build on. Most of METIS's data is held in two structs, \texttt{graph\_t} and \texttt{ctrl\_t}. The former contains the graph itself and all the structures required to represent it, while the latter holds the information METIS needs to run, such as memory allocation requirements and the number of vertices in the coarse graph. We added two new structs to METIS: one to carry per-partition information, and one for the gain table (see Section~\ref{Partition specific gain calculation}). The \texttt{pinfo\_t} struct holds an integer representing a partition's outgoing volume, and \texttt{graph\_t} now carries a pointer to an array of these. At the start of refinement we allocate $k$ \texttt{pinfo\_t} entries, one per partition, and update them throughout refinement to track partition-specific volumes. The second struct, \texttt{vol\_refinement\_table}, holds a pointer to an integer array called \texttt{gain\_list}; \texttt{graph\_t} in turn stores a pointer to an array of \texttt{vol\_refinement\_table} structs. Together, these form the two-level gain table described in Chapter~\ref{cv_calculation}. We opted for this structure over a plain double pointer for readability: it makes the per-partition grouping of the gains explicit at the type level.

When refinement begins, METIS allocates all the memory needed by its refinement structures. NVOL reuses these structures and adds the two described above. The per-partition information requires $k$ integers, while the refinement table requires $k \times k$ integers. Allocating a full table for every boundary vertex would be prohibitively expensive, so we instead allocate a single shared refinement table that is repopulated for each vertex under consideration. This keeps NVOL's memory overhead only marginally higher than that of METIS with VOL.

Once allocated, these structures must be refreshed each time the graph is projected to a new level. During the preparation phase at each level, METIS zeroes out its refinement structures and recomputes them. The graph is projected up to a finer level, and all the structs containing information about neighboring vertices, partitions, and edges must be updated. We added an NVOL branch in all relevant preparation dispatch functions. Most of the existing code was reusable, since NVOL shares the same base as VOL. When evaluating the external degree and the set of neighboring partitions of each vertex, we now also store the needed information in \texttt{pinfo\_t}. The cost of these additions is negligible, since the information was already being computed and was simply not retained. For each visited vertex, we increment its partition's outgoing volume by the vertex's size once per unique neighboring partition.

With the data structures and setup work in place, we now turn to the refinement algorithm itself. The heart of NVOL lives in \texttt{kwayfm.c}, which contains all the refinement algorithms. The file's dispatching function, \texttt{Greedy\_KWayOptimize}, selects a refinement algorithm based on the objective function; we added a new branch that dispatches to our refinement function, \texttt{Greedy\_KWayNVolOptimize}. NVOL's implementation rests on two functions: \texttt{Greedy\_KWayNVolOptimize} and \texttt{compute\_potential\_move\_volume\_change}. The former iterates over the boundary set to decide which moves to perform; it is adapted from METIS's existing \texttt{Greedy\_KWayVolOptimize}, retaining the overall priority-queue-driven structure but replacing the move-selection logic to target the NVOL objective function. The latter is a new helper that supplies the per-partition volume gain calculations (see Section~\ref{Partition specific gain calculation}) required by the modified selection logic. Each is described in detail below. 

\subsubsection{Greedy\_KWayNVolOptimize}
\texttt{Greedy\_KWayNVolOptimize} follows the same outer structure as the other greedy refinement algorithms: boundary vertices are inserted into a priority queue ordered by the objective function's value, extracted one at a time, and moved accordingly. Using partition-specific gain to order the queue would have required either storing a full table for every vertex or recomputing it twice, once when ordering and once during refinement. To avoid this duplication, we chose to order the queue by the total communication volume gain, which was already available and equals the sum of the partition-specific gains. NVOL diverges from METIS in how it chooses the destination partition.

The algorithm then iterates through the priority queue, extracting the top vertex on each pass. NVOL strictly follows the description in Section~\ref{sec:NVOL refinement}, and its implementation closely mirrors Algorithm~\ref{alg:ref_nvol}. Each time a vertex is extracted, NVOL populates the gain table using \texttt{compute\_potential\_move\_volume\_change} (described below) and then iterates over all possible destinations to find the most viable option. Each iteration follows the same pattern: identify the highest local volume that would result from the move, then evaluate the move itself. Since only the neighbor partitions of the extracted vertex experience a change in volume, an inner loop inspects only those. Within this loop, the relevant gain values in the gain table are subtracted from the corresponding current partition volumes stored in \texttt{pinfo\_t}.

After the inner loop, NVOL evaluates the candidate move through a series of conditional checks. If no move has yet been accepted, we simply require that the highest local volume after the move be smaller than the highest recorded volume in the graph. This value is stored in the \texttt{graph\_t} struct and updated within \texttt{compute\_potential\_move\_volume\_change}. This guarantees that any accepted move will not increase the graph's maximum per-partition outgoing volume. Once an initial candidate has been accepted, we look for the move that yields the smallest maximum local volume. Ties are broken in two stages: we first prefer destinations toward which the vertex has the greater external degree, and if those are also tied we prefer moves toward the more underweight partition. METIS's own structures already supplied all the information needed about degree and weights.

Once every destination has been evaluated and a move selected, we update the \texttt{pinfo\_t} entries accordingly; if no move was accepted, the algorithm simply continues to the next boundary vertex. The remaining machinery (boundary updates, subdomain connectivity tracking, and the rebalancing branch) is left essentially unchanged from the upstream METIS code. This process repeats until the priority queue is empty. The greedy refinement as a whole is repeated a fixed number of times (10 by default); however, if no moves are made or no change in volume occurs, the algorithm exits early.

\subsubsection{compute\_potential\_move\_volume\_change}
\texttt{compute\_potential\_move\_volume\_change} is the helper that makes move selection possible. The function takes three arguments: the \texttt{ctrl\_t} structure, the \texttt{graph\_t} structure, and the id of the vertex under consideration, denoted~$i$.

It implements Algorithm~\ref{alg:ref_nvol}, described in Section~\ref{sec:NVOL refinement}. The function begins by zeroing out the refinement table to ensure no stale values from previous vertices remain. Since this pass already touches each of the $k$ rows of the table, we also use it to record which partition currently has the highest volume; this value serves as the ceiling when choosing moves. The two-level structure of the refinement table behaves like a matrix: the outer index selects a row, and the \texttt{gain\_list} index selects a column. Thus \texttt{vol\_refinement\_table[p].gain\_list[j]} stores the volume change that partition $p$ would experience if vertex $i$ were moved to partition $j$. To fill this table, the function walks the neighborhood of $i$ and updates the relevant entries according to the first three cases described in Section~\ref{Partition specific gain calculation}. Once all neighbors have been accounted for, the function handles the fourth case and the moving vertex's own volume contribution in an additional loop, adjusting the source and destination partition volumes accordingly.

Although the helper is conceptually simple, this per-vertex work is the dominant source of NVOL's runtime overhead, which the next subsection examines.

\subsubsection{Time complexity}
The additional work required by NVOL comes at a measurable cost to METIS's runtime. The two parameters that drive this slowdown are the average vertex degree and the number of partitions. The added iterations walk over both the neighboring vertices of each visited vertex and the partitions adjacent to it, so denser graphs and finer partitionings each independently inflate the work performed per refinement step. When both quantities are large simultaneously, the effects compound and the runtime penalty becomes substantial.

The source of this overhead becomes clear when we isolate the portion of the code where \texttt{Greedy\_KWayNVolOptimize} and \texttt{Greedy\_KWayVolOptimize} diverge. Central to the NVOL variant is the call to \texttt{compute\_potential\_move\_volume\_change}. This routine has a time complexity of $O\left(|N_i|(|N_i| + |V^{\mathrm{adj}}_i|)\right)$, where $|N_i|$ denotes the number of neighboring partitions of a vertex $i$ and $|V^{\mathrm{adj}}_i|$ the number of neighboring vertices of $i$. Because this work is repeated for every vertex considered during refinement, even a modest constant-factor difference at the per-vertex level translates into a significant aggregate slowdown over a full pass. The same part of code in VOL, on the other hand, has a complexity of $O(|N_i|)$, simply iterating once over the neighboring partitions of the vertex.

For the smaller graphs there is no difference, but for most graphs NVOL takes on average 7-11 times more than VOL, with the biggest graphs at high partition counts being up to 60 times slower. During refinement, every visited vertex triggers a set of additional iterations that are not present in the VOL implementation of METIS. These extra passes accumulate quickly as the graph grows. The gap widens as the input size increases.

Having described how NVOL is implemented and at what cost it runs, we now turn to the Python tooling built around it for analyzing and visualizing the partitions it produces.

\section{Graph tools} \label{sec:graph tools}
In this section we present a few Python tools developed during this thesis to display and analyze the results produced by the partitioning algorithm, as well as to format graphs. All scripts are available in a public repository\footnote{\url{https://github.com/riccardoie/Graph-tools/tree/main}}. Each subsection presents one tool.

\subsection{translate\_to\_metis}
All graphs used in this thesis were in Matrix Market exchange format (.mtx) and therefore needed to be translated into a format METIS could use. The .mtx format is a plain-text format for representing sparse and dense matrices; when the matrix encodes a graph's adjacency, each non-zero entry corresponds to an edge. After the comment lines (prefixed with \texttt{\%}) and a header, each subsequent line lists the two endpoints of an edge: for example, \texttt{1 24} denotes an edge between vertex 1 and vertex 24. Our script parses the file, discards self-loops, and stores each remaining edge as a sorted tuple in a set, which deduplicates the entries. It then builds an adjacency list by iterating over the edge set and, for each edge $(u, v)$, appending each endpoint to the other's neighbor list. Since Matrix Market files are 1-indexed while Python lists are 0-indexed, we offset the list indices by one while keeping the vertex ids themselves 1-indexed, as required by the METIS format. Once all edges have been processed, the adjacency list is written as a METIS graph file.

\subsection{calc\_partitions}
To verify and analyze the per-partition outgoing communication volumes produced by our experiments, we wrote a small program, \texttt{calc\_partitions}, that computes them directly from the partition files. The program contains a function, \texttt{outgoing\_volstats}, which takes two arguments: the original graph filename and the path to the folder containing the partition files. Given these inputs, the function computes the volumes for every \texttt{.part.$k$} file associated with that graph. The results are written to a text file divided into sections, one per partition file, each introduced by a header of the form \texttt{graphName.part.$k$}. Within each section, the partitions are listed in order of partition id, from $0$ to $k-1$.

For every partition file, the function initializes an array of length $k$ and then iterates over the graph's vertices, collecting for each vertex the set of distinct partitions among its neighbors that differ from its own; \texttt{outgoing\_volstats} adds the size of this set to the entry for the vertex's own partition. Using a set ensures each external partition is counted at most once per vertex, matching the definition of communication volume from Section~\ref{Partition specific gain calculation}. Once the graph has been fully traversed, the resulting array is stored in a dictionary keyed by \texttt{graphName.part.$k$} and written to the output file.

\subsection{form\_statistics}
To aggregate the per-partition volumes computed by \texttt{calc\_partitions} without recomputing them, we wrote the script \texttt{form\_statistics}, which processes the text files produced by that program. For each input file, \texttt{form\_statistics} reads the contents line by line and parses two types of entries: lines containing \texttt{.part.}, treated as headers marking the start of a new partition file's section, and lines starting with \texttt{Pid}, each containing the outgoing volume of the corresponding partition. 

The values are grouped by section in a dictionary, and for each group we compute the total, maximum, average, and standard deviation of the per-partition outgoing volume. The results are written to a CSV file, one per input file.

\subsection{graph\_results\_vol}
To evaluate the distribution of per-partition outgoing volumes, we wrote a simple script that plots them for a given partition file. The script takes three inputs: the graph file, the number of partitions, and an identifier specifying which partition file folder to read. It reads the file produced by \texttt{calc\_partitions} and collects all values from the section matching the requested number of partitions. For comparison, the script also reads the corresponding VOL volumes from a hardcoded path; the two sets of volumes are then plotted side by side as a bar chart.

\subsection{stats\_vol}
\texttt{stats\_vol} reads the files produced by \texttt{form\_statistics} and plots their results. It takes a graph filename and an identifier specifying which partition file folder to read as input, and compares the resulting statistics against the corresponding VOL reference values. It produces four plots: the total outgoing communication volume, the maximum partition volume, the percentage difference in maximum partition volume between NVOL and VOL, and the standard deviation of the partition volumes.

\subsection{model\_estimates} \label{sec:model estimates}
\texttt{model\_estimates} estimates the resulting per-partition communication time induced by a partitioning. We use these values to compare how NVOL's partitions perform against VOL's; reducing the communication time should let the partitioned application finish faster. The formulas implemented in this program are taken from~\cite{Xing_of_Heterogeneous}. They estimate the resulting per-process communication time while accounting for the network bandwidth contention caused by competing Message Passing Interface (MPI) processes. We specifically use Equation~\ref{eq:formula8} from~\cite{Xing_of_Heterogeneous}, which models processes with multiple neighbors and non-uniform message sizes. In this model, each process is assigned a partition and is symmetrically connected through a network whose accessible bandwidth is reduced by contention among the processes.

To understand the program, a few definitions are needed. All calculations in the model are based on the incoming communication volume of a partition. Whereas outgoing communication volume measures the data a partition must send out, incoming communication volume measures the data it must receive from outside. Using the notation from Chapter~\ref{cv_calculation}, the incoming communication volume of partition $n$ is:
\begin{equation}\label{eq:incoming vol}
	\mathrm{incoming~volume}[n] \;=\; \sum_{\substack{v \in V_b \\ n \in N_v}} w(v)
\end{equation}
where $V_b$ is the set of boundary vertices and $N_v$ is the set of neighboring partitions of $v$, excluding $v$'s own partition $P[v]$. In other words, every boundary vertex $v$ contributes its weight $w(v)$ to the incoming volume of partition $n$ if $n$ is one of $v$'s neighboring partitions. Each process is assigned a partition, so the incoming volume $V$ of a process is the incoming volume of its assigned partition.

To compute the estimates as described in~\cite{Xing_of_Heterogeneous}, we sort the incoming volumes in ascending order, $V_0 \leq V_1 \leq \dots \leq V_{N-1}$. For each process $i$ we also record the number of its neighboring processes $M^{\mathrm{in}}_i$ and the message size $s_{i,j}$ sent from process $j$ to $i$. The formulas additionally require a start-up latency $\tau$ and a network bandwidth array $BW$. With these in hand, we compute:
\begin{equation}\label{eq:formula8}
	T_i = M^{\mathrm{in}}_i \cdot \tau + \max\left(t^{\mathrm{recv}}_{i},\; t^{\mathrm{send}}_{i,0},\; t^{\mathrm{send}}_{i,1},\; \dots,\; t^{\mathrm{send}}_{i,M^{\mathrm{out}}_i-1}\right).
\end{equation}
The model estimates the resulting communication time for process $i$ based on two components: the latency of sending messages, and the longest time it takes to finish communicating with other processes. The latter is computed by estimating both the time to receive all incoming messages and the time to deliver all outgoing messages, then taking the maximum. The time process $i$ takes to receive its messages is defined by Equation~\ref{eq:formula7}:
\begin{equation}\label{eq:formula7}
	\begin{aligned}
		t_0^{\mathrm{recv}} &= \frac{N \cdot V_0}{BW_{MP}(N)}, \\
		t_i^{\mathrm{recv}} &= t_{i-1}^{\mathrm{recv}} + \frac{(N - i)\,(V_i - V_{i-1})}{BW_{MP}(N - i)}, \quad i = 1, 2, \dots, N-1.
	\end{aligned}
\end{equation}
To compute the communication time it takes to deliver a message, $t^{\mathrm{send}}_{i,j}$, the calculation is reversed: the time for $i$ to send to $j$ equals the time for $j$ to receive from $i$. Equation~\ref{eq:formula9} shows how this is computed; the incoming messages $s_{i,j}$ are ordered by value, $s_{i,0} \leq s_{i,1} \leq \dots \leq s_{i,M^{\mathrm{in}}_i - 1}$:
\begin{equation}\label{eq:formula9}
	\begin{aligned}
		t_{i,0}^{\mathrm{recv}} &= \frac{M^{\mathrm{in}}_i \cdot s_{i,0}}{V_i} \cdot t_i^{\mathrm{recv}},\\
		t_{i,j}^{\mathrm{recv}} &= t_{i,j-1}^{\mathrm{recv}} + \frac{(M^{\mathrm{in}}_i - j)\,(s_{i,j} - s_{i,j-1})}{V_i} \cdot t_i^{\mathrm{recv}}, \quad j = 1, 2, \dots, M^{\mathrm{in}}_i-1.
	\end{aligned}
\end{equation}
For a more detailed description of these formulations, we refer the reader to~\cite{Xing_of_Heterogeneous}.

With the formulations in place, we can turn to the implementation. Our program takes four arguments: the graph, an identifier specifying which partition file folder to read, the number of partitions, and the startup latency. The network bandwidth values are hardcoded in the program; however, they could easily be exposed as an argument later. Both the startup latency and the network bandwidth values were kindly provided by the authors of~\cite{Xing_of_Heterogeneous}. These bandwidth values describe a realistic example of the accessible network bandwidth as a function of the number of competing processes.

The program begins by computing the incoming communication volume for each process. From these volumes, together with the network bandwidth and latency arguments, we estimate how long each process takes to complete its communication step. The formulas operate on the incoming volumes sorted from lowest to highest and evaluate the receive and send times for each process: Equation~\ref{eq:formula7} estimates the total communication time a process spends receiving messages, while Equation~\ref{eq:formula9} gives the time to send a message to a specific destination.

Equation~\ref{eq:formula8} then combines these results: it takes the maximum of the receive time and the individual send completion times, and adds the startup latency multiplied by the number of neighboring processes. The idea behind taking the maximum time is that a process can only progress as fast as its slowest communication operation. The latency captures the fact that every message exchange incurs a fixed startup cost regardless of the volume transferred. All three formulas are applied to every process, and the results are plotted as a bar chart.

For comparison, the program calculates the estimate on two partitionings: the one produced using NVOL and the one produced using VOL. Both sets of results are shown in the same chart for easy comparison. In Section~\ref{sec:model estimates results} we look at some of the charts produced by \texttt{model\_estimates}.
